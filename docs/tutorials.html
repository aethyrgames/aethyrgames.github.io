Aethyr · your AI BP_Enemy (parent: Character), 5 variables, 2 functions. BeginPlay seeds health and starts a patrol Timeline; OnPerceptionUpdated switches to a chase state and calls FireWeapon. Here's the event graph topology…
used list_blueprints → summarize_blueprint → get_graph

Want a whole folder at once? Hand it to the audit agent:

youAudit /Game/AI: which Blueprints override BeginPlay, and are any variables unused?
bp-audit agent Scanned 23 Blueprints. 6 override BeginPlay; BP_Turret has 3 variables never read; BP_Spawner references a deleted actor class…
a read-only fan-out: nothing was modified

Reads can't change anything, so this is the safest way to start. No need to enable writes. Great for code review and onboarding.

③ All or nothing the safety one

Renames and restructures are where things break. Make the AI do it as a single transaction that either fully lands or fully reverts.

youRename Health to CurrentHealth, add a MaxHealth float, and clamp damage against it, all as one atomic change. Preview it first.
Aethyr · your AI Here's the dry run: 1 rename, 1 new variable, 2 rewired nodes, 1 clamp inserted, no conflicts. Apply it?
used preview_ops (nothing written yet)
youApply.
Aethyr · your AI Applied as one batch, compiled, saved, backed up. If the clamp had failed to compile, the rename would have rolled back too, so your graph is never left half-done.
used apply_ops (atomic:true) → rolled-forward, backup written

Why preview? preview_ops is a read-only dry run: it tells you exactly what a batch will touch before a single byte is written. The bp-batches skill teaches the AI to always offer it for multi-step edits.

④ When the ritual fails

Errors happen. With the bp-debug skill, they're legible instead of cryptic.

youThat returned AssetNotFound: /Game/AI/BP_Enemy.BP_Enemy. What went wrong?
bp-debug · your AI You passed an object path; Aethyr addresses assets by package path. Drop the .BP_Enemy suffix → /Game/AI/BP_Enemy. Re-running now… found it, 5 variables. (If a name's just close, the error lists the real ones so you can pick.)
no asset touched: failed calls are side-effect-free

Every error names what it tried, why it failed, and the closest valid value. A failed write makes no change, so you can always just fix the input and re-ask.

youLooks right. Commit it.

Under the hood: the whole handler went in as one apply_ops batch wrapped in a single undo transaction. Either every node and wire lands together or none does, so the graph never sits half-built between calls.

Look before you leap: preview_ops renders the exact changes without writing a byte, and compile_function assembles the graph from plain pseudocode so you confirm it builds before it ever reaches the asset. Reads like get_compile_status are free to call as often as you like within a turn.

Safety net: every write passes a compile gate and copies the existing .uasset to a timestamped backup before the first save of the session. If a change goes wrong, restore the backup over the file while no editor holds it. Reads never mutate anything.

used list_variables
youRight. Grow it into a real system: rename Health to CurrentHealth, add a MaxHealth float that's editable, replicated, default 100, and a TakeDamage(Amount) function that subtracts the damage and clamps CurrentHealth to the [0, MaxHealth] range. File both vars under a Vitals category. And do it as one move, I don't want a half-finished asset.
bp-batches · your AI Staged as a single batch, nothing written yet. Let me dry-run it first so you can see exactly what it touches.
used preview_ops
Aethyr · your AI The preview comes back clean: 1 rename (Health → CurrentHealth, all 0 existing references rewritten), 1 new variable (MaxHealth, float, EditAnywhere + replicated, default 100), 1 new function (TakeDamage with an Amount input, body subtracts then clamps to [0, MaxHealth]), and both vars filed under Vitals. No conflicts, no name collisions, no dangling pins.
used rename_variable · add_variable · add_function → add_function_input → add_node → wire_pins · set_variable_category ×2
youThat's exactly it. Commit it.
bp-batches · your AI Applied as one atomic transaction. The rename, the new MaxHealth, the TakeDamage function, and both category moves landed together. TakeDamage compiled clean, so the whole batch held; had the clamp failed to build, the rename would have reverted with it and you'd be back at the bare float, untouched. Verified after: BP_Enemy is compile-clean, no errors, no warnings.

Why preview first: preview_ops walks the entire batch against the live Blueprint and reports what it would do, the rename count, the new symbols, any collision, without writing a single byte. You approve a plan, not a guess.

All or nothing: apply_ops runs the batch as one transaction behind the compile gate. If the TakeDamage clamp had failed to compile, the rename and the new variable roll back with it, so the asset is never stranded between a bare float and a finished system. A backup is written before the first save, either way.

Reads are free, so look first: list_variables never mutates and costs you nothing within a turn. Confirming the starting shape before you plan a change is the cheap habit that makes the expensive step safe.

⑧ Fork a proven Blueprint into a variant no copy-paste

You have a working BP_Enemy and you want a meaner sibling, BP_EliteEnemy, that shares every line of logic but hits harder.

youMake a new Blueprint called BP_EliteEnemy with the same parent as BP_Enemy. I want a variant, not a fresh actor.
Aethyr · your AI Read BP_Enemy first: its parent is ACharacter. Created BP_EliteEnemy under /Game/AI/ on that same parent, so it inherits the identical component skeleton. Empty graphs for now. Compiled clean.
used read_blueprint → create_blueprint
youNow bring the brains across. Same functions as BP_Enemy, wired exactly the same. No hand-copying nodes.
Aethyr · your AI Listed the functions on BP_Enemy: TakeHit, ChooseTarget, and OnDeath. Declared matching stubs on the elite, then cloned each graph across whole, so every node, pin default, and wire came over intact, not redrawn. The event graph came too. Re-compiled the elite: clean, zero dangling pins.
used list_functions → add_function ×3 → clone_graph ×4 → compile_function
youGood. Now make it elite: 250 health instead of 100, and 40 damage instead of 25. Just the numbers.
Aethyr · your AI Touched only two defaults on the variant: MaxHealth 100 → 250 and AttackDamage 25 → 40. No graph edits, no structural change. Backed up the asset before the first write, then saved through the compile gate.
used set_variable_default ×2 → compile_function
youProve it. I want to see that the only differences from BP_Enemy are those two numbers, nothing structural snuck in.
Aethyr · your AI Ran a structural diff between the two Blueprints. Graphs, functions, components, and interfaces are identical, node for node and wire for wire. The diff reports exactly two deltas, both default-value: MaxHealth (100 vs 250) and AttackDamage (25 vs 40). Nothing structural drifted. Your variant is a true fork.
used bp_structural_diff

Why clone, not copy: clone_graph moves a graph as one unit, carrying every wire and pin default with it, so the elite's logic is the proven logic, not a re-drawing you have to re-verify. Hand copy-paste is where a missed connection hides.

The diff is your proof: bp_structural_diff is read-only and never touches either asset, so running it as the final step costs nothing and turns "I think they match" into "the only deltas are these two defaults." Reads are cheap within a turn, so verify freely.

If the elite had failed to compile after the stat change, the compile gate would have refused the save and the pre-write backup would still hold the last good copy, so a bad variant can never overwrite your working BP_Enemy or leave the elite half-written.

The audit you just did and the edit that follows are two deliberate, separate acts.

⑩ When a batch goes sideways recovery

A five-step edit trips on step four, and the whole thing rolls back untouched, exactly the safety net you want.

youBuild out BP_Turret: add a TargetActor variable, a FireRate float, an AcquireTarget function, then wire its found-actor pin into a new SetTargetActor call. Do it as one batch so it's all-or-nothing.
Aethyr · your AI Staged it as a single atomic apply_ops. Steps 1 through 3 applied, but step 4 failed the compile gate, so the whole batch was rejected. The error is pinned to that op: wiring FoundActor (Actor Object Reference) into the NewTarget input (Pawn Object Reference) is a type mismatch, the pins don't connect.
used apply_ops → get_compile_status
youSo nothing got written to the asset?
bp-debug · your AI Correct, nothing. Atomic means the whole batch lands or none of it does, and step 4 failed, so the save never happened. I re-read BP_Turret to confirm: no TargetActor, no FireRate, no AcquireTarget on disk. The Blueprint is byte-for-byte what it was before you asked. Reads like that are free inside the turn, so confirming costs nothing.
used read_blueprint → get_dirty_packages
youFix the one bad op, then check it before you commit anything.
Aethyr · your AI The fix is to cast first: I retargeted step 4 to feed AcquireTarget's output through a Cast To Pawn node, whose As Pawn pin (Pawn Object Reference) matches SetTargetActor's input. Then I dry-ran the whole five-step batch with preview_ops: it validates clean now, every pin type lines up, nothing touched the asset yet.
used preview_ops
youGood. Apply it for real.
Aethyr · your AI Done. apply_ops ran all five steps and saved. A backup of BP_Turret went to Saved/AethyrBackups/ before the first write, and the final get_compile_status came back clean, zero errors, zero warnings. The variables, the AcquireTarget function, and the cast-and-set wiring are all in.
used apply_ops → get_compile_status

Why this is safe: a batch apply_ops is atomic. Every op runs against an in-memory copy and the result must pass the compile gate before a single byte hits disk. One bad op fails the whole batch, and the on-disk asset stays exactly as it was, no half-written graph to clean up.

The loop that pays off: when a step fails, the error names the offending op and both pin types, so you fix one thing, not the batch. Re-run preview_ops to validate with zero writes, then apply_ops once it's green. The bp-debug skill walks this read, fix, re-preview loop, and reads stay cheap the whole way.
